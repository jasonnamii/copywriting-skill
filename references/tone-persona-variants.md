# Creator Persona Sub-types

"Creator" is not monolithic. Different creator personas have different sensibilities, pain points, and what they consider "hip."

---

## The 4 Creator Personas

### 1. Designer (Visual Thinker)
**Who:** Figma/Framer users. Design system builders. 3-5 years experience. Perfectionist.

**Sensibility:** Minimal, typographic, Swiss-grid. Trusts white space. Skeptical of emoji and casual language.

**Pain Points:**
- Handoff from design to dev is messy
- Design systems get ignored in code
- Pixel perfection vs. code reality gap
- Time lost in back-and-forth with engineers

**Hip Adjustment (4 NN/g Dimensions):**
- Funny: Serious (stays neutral, dry humor only)
- Casual: Formal (respects their taste, uses proper language)
- Irreverent: Respectful (trust the craft)
- Enthusiastic: Matter-of-fact (tone of craft mastery)

**Design-Specific Hip Examples (Korean):**
1. "디자인 시스템을 코드로. 1:1 매칭. 픽셀 손실 없이."
2. "설계하면 된다. 우리가 구현 규칙을 지킨다."
3. "8-point grid. 색상 정확도. 타이포그래피 구조. 모두 보존됩니다."

**Design-Specific Hip Examples (English):**
1. "Design system to code. Pixel-perfect fidelity."
2. "You design. We respect your decisions."
3. "Type scales, color systems, spacing ratios—all intact."

**Anti-patterns (Alienate Designers):**
- ❌ Using emoji excessively ("Design ✨ to ✨ Code ✨")
- ❌ Oversimplifying craft ("Make it easier")
- ❌ Forcing playfulness ("Let's vibe together")
- ❌ Dismissing their process ("Skip the design review")
- ❌ Corporate speak ("Seamlessly integrate")

### 2. Developer (Builder)
**Who:** Linear/Vercel users. Full-stack builders. Pragmatic. Velocity-focused.

**Sensibility:** CLI-inspired, precise, no fluff. Values speed and accuracy. Reads documentation obsessively.

**Pain Points:**
- Badly exported code from designers
- Figma-to-code workflow is manual
- Dependencies bloat
- Integration with CI/CD is hard
- Code review of designs is tedious

**Hip Adjustment (4 NN/g Dimensions):**
- Funny: Serious (only self-deprecating or industry humor)
- Casual: Formal (technical precision valued)
- Irreverent: Respectful (of best practices)
- Enthusiastic: Matter-of-fact (results speak)

**Developer-Specific Hip Examples (Korean):**
1. "Figma 파일이 git-ready 코드가 됩니다. No cleanup."
2. "API-first. 당신의 CI/CD에 잠금장치처럼 끼웁니다."
3. "의존성이 없어. 프로덕션 코드, 바로. Zero technical debt."

**Developer-Specific Hip Examples (English):**
1. "Design files to git-ready code. No cleanup needed."
2. "API-first. Drops into your CI/CD."
3. "No dependencies. Production-ready. Zero debt."

**Anti-patterns (Alienate Developers):**
- ❌ Vagueness ("Speeds up development")
- ❌ Dismissing their toolchain ("We handle it all")
- ❌ Non-technical language ("Magical automation")
- ❌ Corporate case studies ("Increase productivity")
- ❌ Cutesy tone ("Let's make coding fun")

---

### 3. Video Creator (YouTube/TikTok)
**Who:** YouTube, TikTok, Instagram. Motion-first. Community-building focus. Trend-aware.

**Sensibility:** Energetic, meme-literate, emoji-comfortable. Responds to social proof. Values community.

**Pain Points:**
- Need to iterate fast on thumbnails, thumbnails, thumbnails
- Thumbnail design → export pipeline is slow
- Thumbnail A/B testing takes forever
- Feedback from audience is scattered
- Building a portfolio is time-consuming

**Hip Adjustment (4 NN/g Dimensions):**
- Funny: Funny (relatability wins)
- Casual: Casual (speaks their language)
- Irreverent: Irreverent (challenges norms)
- Enthusiastic: Enthusiastic (energy is good)

**Video Creator-Specific Hip Examples (Korean):**
1. "썸네일 50개? 하루에 끝내. 30초마다 1개."
2. "당신의 커뮤니티가 본다. 그들 피드백을 C8 안에서 본다."
3. "Figma 영상 생성 → 유튜브 업로드. 한 번에."

**Video Creator-Specific Hip Examples (English):**
1. "50 thumbnails a day. One every 30 seconds."
2. "Your community sees it. Their feedback matters. It's all here."
3. "Design → export → YouTube in one flow."

**Anti-patterns (Alienate Video Creators):**
- ❌ Treating them as "casual" ("Easy tool for everyone")
- ❌ Boring language (they want energy)
- ❌ No community references ("You, alone, can do this")
- ❌ Slow promises ("Over time, you'll see benefits")
- ❌ Gatekeeping language ("For designers only")

---

### 4. Writer/Artist (Are.na/Substack)
**Who:** Are.na users, Substack writers, visual artists. Conceptual. Meaning-focused.

**Sensibility:** Literary, poetic, whitespace-as-meaning. Skeptical of tools overselling efficiency. Values intention and craft slowness.

**Pain Points:**
- Turning written concepts into visual form is manual
- Mood boards don't export to design files
- Illustration→→web publication pipeline is fragmented
- Too many tools, not enough meaning
- Tools should disappear; only the work should remain

**Hip Adjustment (4 NN/g Dimensions):**
- Funny: Serious (dry, conceptual humor only)
- Casual: Formal (respects intellectual tone)
- Irreverent: Respectful (honor the intention)
- Enthusiastic: Matter-of-fact (understatement is powerful)

**Writer/Artist-Specific Hip Examples (Korean):**
1. "당신의 아이디어. 우리는 보이는 형태를 입혀줄 뿐."
2. "도구는 사라져야 해. 당신의 작품만 남아야 해."
3. "의도에서 코드까지. 중간을 생략하지 않으면서도."

**Writer/Artist-Specific Hip Examples (English):**
1. "Your idea. We just give it form."
2. "Tools should disappear. Only the work remains."
3. "From intention to code. Nothing skipped."

**Anti-patterns (Alienate Writers/Artists):**
- ❌ Overselling speed ("Ship 10x faster")
- ❌ Reducing to metrics ("Increase productivity")
- ❌ Playfulness ("Make creativity fun")
- ❌ Casual energy (they prefer contemplation)
- ❌ Forced community ("Everyone's doing it")

## Anti-patterns by Persona

### What Alienates Designers
- Treating design as decoration ("Just make it pretty")
- Dismissing the design system ("We'll handle it")
- Forced casualness ("Let's vibe with the pixels")
- Oversimplifying ("Easier is better")
- Not respecting craft ("One-click magic")

### What Alienates Developers
- Vagueness ("Magically faster")
- Technical inaccuracy ("We do all the hard stuff")
- No documentation ("Just feel your way through")
- Forcing playfulness ("Coding is fun now!")
- Hiding trade-offs ("This works for everything")

### What Alienates Video Creators
- Slow promises ("Over time, results will come")
- Gatekeeping ("For professionals only")
- Ignoring community ("You, alone, ship faster")
- Energy depletion ("Take your time, steady wins")
- Missing social proof ("Nobody else uses this")

### What Alienates Writers/Artists
- Metrics obsession ("50% faster = better")
- Tool proliferation ("Add another app to your flow")
- Forcing efficiency ("Stop overthinking, ship")
- Casual dismissiveness ("Just use templates")
- Treating intention as secondary ("Get it out fast")

## Persona-Mixing Strategy

Not all creators fit one bucket. Some are **Designer-Developer hybrids**. Some are **Developer-Video Creator**.

**Decision Rule:**
1. Identify **primary persona** (where they spend 70% of their time)
2. Honor primary persona's tone
3. Reference secondary persona's pain points in social proof / testimonials
4. Never mix tones in the same paragraph (confusing)

**Example - Designer-Developer Hybrid:**
- Primary: Designer (formal, respectful tone)
- Secondary: Developer (no technical debt reference)
- Copy: "Design specifications preserved. Code is production-ready. No cleanup needed."

## Testing Persona-Specific Copy

Best practice: A/B test the same message across persona-specific and universal versions.

**Universal (A1 Feature Overview):**
"C8 converts designs to code in minutes."

**Designer-Specific:**
"Design systems to code. Fidelity preserved. Your decisions respected."

**Developer-Specific:**
"Figma to GitHub in one step. No cleanup."

**Video Creator-Specific:**
"Design once. Export infinitely. A/B test everything."

**Writer-Specific:**
"Your concept becomes visible. Nothing lost in translation."

Test which resonates. Data drives tone as much as intuition.
